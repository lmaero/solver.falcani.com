# The falcani's solver framework

## Luis (falcani's founder) feedback and analysis

The current solver framework:
1. Is quite rigid and does not allow for easy integration of new solvers or modifications to existing ones. This can be a significant barrier to innovation and improvement in the solver ecosystem.
2. It has a lot of opinionated choices that can lead to suboptimal solutions. For example, the UI-related libraries are like a strength shirt that can affect creativity and flexibility of the solvers. This can limit the ability of developers to create innovative and effective solvers that are tailored to specific problem domains.
3. I haven't tested yet for starting from scratch projects, but I think it's a good idea to have a solid foundation for future projects, that's why I'm proposing improvements to the current framework.
4. The @setup.sh script should not ask for dependencies because every project could have its own stack, and this is related to the next point.
5. Dependencies are not something the framework should care about, it should be a concern of the specific project. The framework should provide a way to easily integrate with different dependency management tools, but it should not dictate which ones to use. This allows for greater flexibility and adaptability to different project needs and preferences.
6. What the framework should do is provide a solid foundation for creating and maintaining projects that are modular, extensible, and easy to use. This includes providing a clear and consistent structure for projects, as well as tools and resources for testing, documentation, and deployment. By focusing on these core principles, the framework can help to foster a vibrant and innovative solver ecosystem that is able to adapt to changing needs and technologies over time.
7. Brainstorming, spec, plan pipeline sounds too familiar to openspec for Domain Driven Development, and I agree that it's a good idea to have a similar approach. So practices like Domain Driven Development, Test Driven Development, SOLID principles, etc. should be encouraged and supported by the framework to promote best practices and ensure consistency across projects.
8. Architecture patterns for API requests, responses and server actions shapes should be an enforced rule. This will help to standardize the API and ensure consistency across projects. It will also make it easier for developers to understand and work with the API, as they will be familiar with the patterns and conventions used. This can lead to faster development and improved collaboration among developers working on different projects within the solver ecosystem.
9. The usage of domain verticals was one of the most important decisions of the framework because it internalizes a more structured approach to organizing the code and making it easier to understand and maintain.
10. All the UI tasks were a disaster in every project; the framework degraded the experience, making worst the UX/UI, and the development experience. The UI is a crucial part of the solver ecosystem, and it should be treated as such. The framework should provide tools and resources for creating effective and user-friendly UIs, but it should not dictate how they are created or what libraries are used. This allows for greater creativity and flexibility in UI design, which can lead to more engaging and effective solvers that are tailored to specific user needs and preferences. I would let off the UI-related libraries and focus on the core solver functionality.
11. There was a lot of missing features in the project even during migrations the framework was not able to completely transfer existing features. If the framework main goal is to allow orchestration of complete production projects, then it should be able to handle all the features that are needed. This includes features such as authentication, authorization, data storage, and more. By providing a comprehensive set of features and tools, the framework can help to ensure that developers have everything they need to create effective and robust solvers that are ready for production use.
12. During the exploration phase, critical decisions should not be made by the framework, instead the human behind the framework should make the final decisions. The framework should provide guidance and support for the exploration phase, but it should not dictate the direction or outcome of the exploration. This allows for greater creativity and innovation in the development process, as developers are free to explore different ideas and approaches without being constrained by the framework's limitations or biases. By fostering a culture of experimentation and exploration, the framework can help to drive innovation and improvement in the solver ecosystem over time.
13. I saw not a single diagram of the architecture of any of the experiments, so something is not working in the framework related to the creation of these architecture diagrams. Even when it is an existing projects that has no documentation at all, understanding of the codebase and rapid exploration of the codebase should be crucial for the orchestrator to be able to understand the codebase and make decisions of what's missing, where the project is going, etc.
14. TDD is an amazing idea, but the agents are not being enforced to follow the TDD approach.
