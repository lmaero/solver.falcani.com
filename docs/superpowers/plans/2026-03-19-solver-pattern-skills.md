# Pattern-Based Skills Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Create 8 pattern-based skills that teach engineering patterns without prescribing specific libraries. Each skill is a SKILL.md file with YAML frontmatter that Claude Code auto-discovers.

**Architecture:** Skills are markdown files with YAML frontmatter installed into `.claude/skills/` by `solver init`. Each skill teaches a pattern (the HOW) independent of any library (the WHAT). The `solver init` command already creates the `.claude/skills/` directory; this plan adds skill content to it.

**Tech Stack:** Markdown with YAML frontmatter. Skills are generated by template functions in the solver CLI's `src/templates/skills/` directory.

**Spec:** `docs/superpowers/specs/2026-03-19-solver-framework-v2-design.md` Section 6.

**Plan scope:** Plan 2 of 3. Depends on Plan 1 (CLI skeleton complete). Plan 3 covers analysis commands.

## File Structure

```
solver/src/templates/skills/
├── action-patterns.ts          # ActionResult pattern, boundary validation, error handling
├── data-access-patterns.ts     # Service layer, validate before writes, typed returns
├── env-validation.ts           # Schema validation at startup, crash if invalid
├── logging.ts                  # Structured logging, levels, redaction
├── ci-cd-patterns.ts           # Quality gates, deploy from main, secrets
├── discovery.ts                # Product-driven questions, WHO/WHAT/WHY
├── domain-patterns.ts          # Domain separation, entities, state machines
└── testing-strategy.ts         # TDD boundary, SRP for testability

solver/tests/templates/skills/
└── skills.test.ts              # Tests for all skill generators
```

Each generator function returns a string containing the full SKILL.md content (YAML frontmatter + markdown body).

### Task 2: Implement action-patterns and data-access-patterns skills

**Files:**
- Create: `solver/src/templates/skills/action-patterns.ts`
- Create: `solver/src/templates/skills/data-access-patterns.ts`

- [ ] **Step 1: Implement action-patterns skill**

Content must cover:
- **Mutation return pattern**: every mutation returns a typed result distinguishing success from failure. The pattern uses a discriminated union (or language equivalent). Concrete shapes shown as pseudocode, not tied to any library.
- **Boundary validation**: every entry point (action, handler, endpoint) validates its inputs against a schema BEFORE any business logic runs. This is defense-in-depth. The schema library is the project's choice.
- **Error handling shape**: errors carry a user-facing message ("We couldn't [action] because [reason]. Try [fix]") plus optional field-level errors for form feedback. Never expose internal details.
- **Non-form mutations**: delete buttons, toggles, status changes also use the same return pattern.

Must NOT reference: react-hook-form, Zod, Next.js Server Actions, shadcn/ui.

YAML frontmatter: name, description, trigger conditions (when the solver is building mutations, forms, or API endpoints).

- [ ] **Step 2: Implement data-access-patterns skill**

Content must cover:
- **Service layer isolation**: all database access goes through service functions, never direct from UI or route handlers. One file per domain entity or collection.
- **Validate before writes**: every write operation validates input against a schema before touching the database.
- **Typed returns**: service functions return typed objects, never raw database documents.
- **Log operations**: every service function logs with a child logger for traceability.
- **Pagination pattern**: reads that return lists include pagination metadata.
- **Index definitions**: indexes are defined as exports, applied via setup script.

Must NOT reference: MongoDB, Mongoose, Prisma, Drizzle, or any specific database.

### Task 5: Implement domain-patterns and testing-strategy skills

**Files:**
- Create: `solver/src/templates/skills/domain-patterns.ts`
- Create: `solver/src/templates/skills/testing-strategy.ts`

- [ ] **Step 1: Implement domain-patterns skill**

Content must cover:
- **Domain separation**: business logic lives in a pure domain layer with zero framework dependencies. No React, no Next.js, no Express, no database imports.
- **Entity design**: entities encapsulate behavior and validation. They are not anemic data holders.
- **State machines**: model entity lifecycle as explicit state transitions. Invalid transitions are caught at compile time or runtime before reaching the database.
- **Domain error hierarchy**: typed error classes that carry semantic meaning (EntityNotFound, InvalidStateTransition, UnauthorizedAction) and map cleanly to HTTP status codes at the boundary.
- **Pure functions, side effects at edges**: domain logic is pure. Side effects (database, network, logging) happen at the infrastructure layer.

Must NOT reference any specific language, framework, or database.

- [ ] **Step 2: Implement testing-strategy skill**

Content must cover:
- **TDD boundary**: what MUST be TDD — business rules, domain logic, calculations, mutations, permissions, data transformations. Write the failing test first, implement minimally, refactor.
- **Tests after implementation**: what gets tested but NOT via TDD — data access queries, route handlers, utility functions. "Port then test" for migrations.
- **No tests needed**: UI components (if they follow SRP and contain no logic), configuration files, static content, seed scripts, dev tooling.
- **SRP as the testing enabler**: if you need to test a UI component's logic, the component is doing too much. Extract the calculation/rule into a pure function and test that. The form captures input and displays output; the function does the work.
- **Test structure**: one test file per source file in domain/application layers. Naming convention: `[name].test.ts` alongside or in a parallel `tests/` directory.
- **What makes a good test**: tests verify behavior, not implementation. Mock at boundaries (external APIs, databases), not within the domain. Each test should be readable as a specification.
